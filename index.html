Task 1 · Grid system

One grid, nine screen sizes

The grid was set up manually in Figma by the lecture rules: documented breakpoints, a consistent size system and layout patterns, with every frame built on auto layout.

Breakpoints

TierFigma framesRange
Desktop1920 · 1440 · 1024≥ 1024 px
Tablet1023 · 860 · 640640–1023 px
Mobile639 · 390 · 320≤ 639 px

Grid principles

  • Desktop: 12 columns, fixed margins, fluid gutters
  • Tablet: 8 columns, tighter margins
  • Mobile: 4 columns, 16 px margins
  • Content never touches the edge — spacing scales down with the tier, not per-frame

Size system

Spacing steps on a small consistent scale (16 / 24 / 32 / 48) reused across padding, gaps and heights — the same values later confirmed during the code rebuild in Task 3.

Layout patterns

3-zone header (left: logo + nav · center: search · right: actions), progressive collapse of secondary elements, wireframe layouts assembled entirely with auto layout — resizing a frame reflows the content instead of breaking it.

Task 2 · Accessibility → spec.md

Audit in, verified spec out

Input

The WCAG 2.2 Level AA audit of the concept was added to the repository: 10 screens, 6 findings (PNP-011…016) and 6 passing patterns that already work correctly.

Process

Claude proposed a change plan first. Every hex value and specification was checked against the exact source quote in the audit file before writing anything. Five changes were applied one at a time, each confirmed before the next began.

Result

spec.md now contains the WCAG 2.2 AA declaration; three confirmed colour tokens (#F6F6F6 · #2C2C2C · #A83232); a contrast-minimums table; a mandatory validation rule for form fields; a full Accessibility Rules section; an updated version history.

Traceability

Every change links to a specific ticket. No values were invented. No existing content was deleted. A summary table maps each finding → spec change → WCAG criterion.

Human verification — checking every update

  • The change plan was reviewed before approval; three values looked like possible AI assumptions and were challenged.
  • Claude was required to quote the exact source line for each: the focus spec (PNP-011), error colour #A83232 (PNP-012), and both neutral tokens (PNP-013 / 015).
  • Quotes were then confirmed manually in the audit file (Ctrl+F: #A83232 — 1/1 match). All three turned out to come from real fix specs — verified, not trusted.

Task 3 · Rebuild from spec.md

Figma → code, refined by dialog

Input

Header.png with 9 responsive variants + a notes file with the exact breakpoint table and visibility rules for every element.

Process

Claude listed its assumptions first (spacing, fonts, breakpoints, icon style) and waited for designer confirmation. Only then it built a single header.html with embedded CSS, applying spec.md rules throughout.

Refinements made through the dialog

  • Header background corrected from dark to white — Claude contradicted its own analysis; caught on first review.
  • Logo mark fixed: the generated SVG read as an “R”, corrected to the proper “P”.
  • Search group repositioned into a true 3-zone flex layout (left / center / right).
  • Overflow collision at 1024–1279 px fixed: fixed-width absolute centering replaced with a flexible min-width 200 / max-width 560 shrink zone.

spec.md rules applied in code

PNP-011 · focus 2px #2C2C2C, lime on dark PNP-014 · aria-label on icon buttons PNP-016 · placeholder #6A6A6A WCAG 2.5.8 · tap targets ≥24px tokens · neutral-900 / teal badge #2F6F73
